# Design: code-explainer — Real-time diff explanations for vibe coders

Generated by /office-hours on 2026-04-13
Branch: master
Repo: code-explainer-gstack
Status: APPROVED
Mode: Builder

## Problem Statement

Non-developers using Claude Code (vibe coders — designers, PMs, founders building real products with AI) accept every AI-generated code change blindly. They cannot read diffs, spot unrelated modifications, or assess security risks. A Stanford study found developers using AI write MORE vulnerabilities AND are MORE confident their code is secure. The false confidence is arguably more dangerous than the vulnerabilities.

Every existing solution works post-hoc: PR summaries (What-the-Diff), code scanning (Checkmarx), governance frameworks. But vibe coders never read PRs. By the time a PR exists, they've already accepted 50 changes they didn't understand.

**The intervention point must be DURING the coding session, not after it.**

code-explainer is a Claude Code post-tool hook that intercepts every file modification, sends the diff to an explanation engine, and prints a formatted plain-language explanation box in the terminal — before Claude continues. Real-time. In-session. At the actual moment of risk.

## What Makes This Cool

1. **Unoccupied niche.** No tool does real-time, in-terminal diff explanations during an AI coding session. The entire security industry built for the wrong moment.
2. **Session drift detection.** Beyond per-change explanations, code-explainer tracks ALL changes across a session and alerts when Claude wanders off-task. "Claude has modified 14 files. 3 are unrelated to your request."
3. **Zero-friction for the audience.** `npx code-explainer init` — one command, interactive setup, auto-detects GPU, pulls the right model, configures hooks. Vibe coders never touch a config file.

## Constraints

- Must work cross-platform: Windows, macOS, Linux, WSL
- Must be stack-agnostic (any language, any framework)
- Must be brownfield-safe (installs into existing projects, never overwrites config)
- Claude Code hooks are synchronous — the explanation adds latency per change
- Ollama engine has no conversation context (only sees the diff)
- Must not require paid API keys for the core experience (Ollama = free)

## Premises

1. Non-developers accept AI code changes blindly — they don't read diffs at all.
2. Synchronous hook is the right UX — Claude waits for the explanation before continuing, even though it adds latency per edit. The pause IS the value.
3. A local 7B LLM can produce useful explanations from just a diff without conversation context. Risk detection is limited to structural/path heuristics.
4. npx distribution works for vibe coders — Node.js is already required by Claude Code.

### Architecture

```
npx code-explainer init
  └─ @clack/prompts interactive CLI
     ├─ Detect OS/environment
     ├─ Choose engine (Ollama / Claude Code)
     ├─ If Ollama: verify install, detect VRAM, pull model
     ├─ Choose detail level (minimal / standard / verbose)
     ├─ Write code-explainer.config.json
     └─ Merge hooks into .claude/settings.json (brownfield-safe)

Post-tool hook flow:
  Claude Code modifies file
    → PostToolUse hook fires (Edit | Write | Bash)
    → Hook script reads stdin (tool payload)
    → Extract diff (git diff for Edit/Write, command analysis for Bash)
    → If Bash: check against capturePatterns filter, skip if read-only
    → Send diff + prompt to chosen engine:
       • Ollama: HTTP POST to localhost:11434/api/generate
       • Claude Code: claude -p with full context
    → Parse response (JSON: { summary, risk, riskReason })
    → Format explanation box → print to stderr
    → Update session state file (/tmp/code-explainer-session-{ppid}.json)
    → If drift detected: print session drift warning box

Session drift detection:
  - Session state: JSON file accumulating { file, timestamp, relatedScore }
  - Drift heuristic (Ollama): file path analysis (auth/payment/env = flag)
  - Drift analysis (Claude Code engine): compare change against original user prompt
  - Session lifecycle:
    • Start: first hook invocation creates session state file
    • Session ID: keyed to Claude Code parent PID (PPID)
    • End: idle timeout (30 min with no hook invocations) or explicit `npx code-explainer summary --end`
    • Cleanup: hooks check for stale session files (>2 hours old) and remove them on startup
  - Three touchpoints:
    1. Threshold alert — prints mid-session when unrelated changes detected
    2. End-of-session summary — `npx code-explainer summary` prints final report and clears session
    3. On-demand — `npx code-explainer summary` (without --end) reads session state without clearing
```

### Explanation Engines

**Ollama (Local LLM):**
- Free, offline, private
- Model tiers (all verified on [Ollama library](https://ollama.com/library)):
  - **≤8 GB VRAM** → `qwen3-coder:30b` (MoE, only 3.3B active params, fits in 8GB)
  - **12-16 GB VRAM** → `qwen2.5-coder:14b` (dense model, strong code understanding)
  - **≥20 GB VRAM** → `qwen2.5-coder:32b` (best local quality)
  - Fallback: `qwen2.5-coder:7b` (smallest dense coder model, works on any GPU)
- Prompt template outputs structured JSON for deterministic formatting
- Risk detection: path-based heuristics (payment files, auth files, env files, config files)
- No conversation context — explains the diff in isolation

**Claude Code (Native):**
- Uses `claude -p` CLI to generate explanation in a one-shot invocation
- **Context limitation:** `claude -p` does NOT inherit the parent Claude Code session's conversation context. To provide context, the hook reads the user's original prompt from the hook payload's `session_prompt` field (if available) or from the most recent `.claude/session.json` state file. If neither is available, falls back to diff-only explanation (same as Ollama, but with a stronger model).
- When context IS available: can detect if a change is unrelated to the user's request (intelligent risk). Session drift detection becomes context-aware, not just path-based.
- When context is NOT available: behaves like a high-quality Ollama engine (better model, same diff-only input).
- Consumes API tokens — the init flow warns about token cost and estimates per-explanation cost (~$0.002-0.01 per explanation depending on diff size).

`code-explainer.config.json` immediately. The user can change multiple settings

**Setting-specific prompts:**

```

  ◉ Write — explains new files
  ◉ Bash — explains destructive commands (rm, mv, git reset, etc.)
```

File exclusions:
```

? Add or remove file patterns:
  ❯ Add a pattern (e.g. *.generated.*)
    Remove a pattern
    Reset to defaults (*.lock, dist/**, node_modules/**)

```

```
  ❯ 5 seconds  — fast, may skip complex changes
    8 seconds  — balanced (recommended)
    15 seconds — patient, rarely skips
    Never skip — always wait for the explanation
```

If the user switches engine from Ollama to Claude Code (or vice versa), the
command handles the transition: verifies the new engine is available, updates
the hooks if needed, and confirms the change.

### VRAM Detection Strategy

**Auto-detection:** NVIDIA only via `nvidia-smi --query-gpu=memory.total --format=csv,noheader,nounits`. If detected, auto-selects the model tier silently.

**Fallback (non-NVIDIA or detection fails):** Present a model chooser with VRAM guidance:

```
? Which model should code-explainer use?
  ❯ qwen3-coder:30b    (recommended for ≤8 GB VRAM — fast, smart, lightweight)
    qwen2.5-coder:14b  (recommended for 12-16 GB VRAM — strong code understanding)
    qwen2.5-coder:32b  (recommended for ≥20 GB VRAM — best local quality)
    qwen2.5-coder:7b   (fallback — works on any GPU, smallest model)
```

The user picks the model directly. No need to know their VRAM number, just pick what runs well on their machine. The default (`qwen3-coder:30b`) is pre-selected because it uses MoE architecture (only 3.3B active parameters despite 30B total), so it runs fast on low VRAM while being smarter than smaller dense models.

### Cold Start Mitigation

Ollama's first inference after model load can take 10-15s (cold start). Mitigations:
- **Loading indicator:** On cold start (first hook invocation), print `[code-explainer] Loading model... (first explanation may take 10-15s)` to stderr before calling Ollama.
- **Warmup command:** `npx code-explainer warmup` sends a trivial diff to Ollama to pre-load the model. Init flow suggests running this after setup.
- **Keep-alive:** Ollama's default keep_alive is 5 minutes. If the gap between edits exceeds this, the next explanation will be slow. Document this behavior.

### Bash Filter — Captured Commands

Only Bash commands that modify the filesystem trigger an explanation:
- File ops: `rm`, `mv`, `cp`, `mkdir`, redirections (`>`, `>>`)
- Package managers: `npm install`, `pip install`, `yarn add`, `pnpm add`
- Permission changes: `chmod`, `chown`
- Git mutations: `git checkout`, `git reset`, `git revert`
- Stream editors: `sed -i`, `awk` (in-place)

Read-only commands (`ls`, `cat`, `grep`, `git status`, etc.) are skipped.

### Key Risk: Prompt Engineering

The subagent correctly identified this as the real product. The prompt template must:
- Output structured JSON `{ summary, risk, riskReason }` for deterministic formatting
- Produce explanations a non-developer understands (no jargon)
- Correctly calibrate risk levels (not too many false positives, not too few alerts)
- Work consistently across languages (TypeScript, Python, Go, Rust, etc.)
- Handle edge cases: empty diffs, binary files, very large diffs (truncation strategy)

### Edge Case Handling (hook level)

- **Binary files:** Detected via `git diff --numstat` binary marker. Hook prints a one-line notice ("Binary file modified: {path}") and skips LLM explanation.
- **Empty diffs:** Hook skips silently (no output, no LLM call).
- **Large diffs (>200 lines):** Truncate to first 150 lines + last 50 lines with a "[...truncated N lines...]" marker. Send truncated diff to engine with a note in the prompt.
- **Skip reason logging:** All skips logged to stderr as `[code-explainer] skipped: {reason}` for debugging.

Plan: test the prompt template against 20+ real-world diffs before shipping.

## Open Questions

1. **Latency budget:** Target < 5s per explanation (success criteria). Cold start may be 10-15s (documented in Cold Start Mitigation). Monitor real-world latency and consider adding a config option to disable explanations for files matching a glob pattern (e.g., skip `*.lock` files).
2. ~~**Large diffs:**~~ RESOLVED — truncate to first 150 + last 50 lines for diffs > 200 lines. See Edge Case Handling.
3. **Session drift — Ollama engine:** Without conversation context, drift detection relies on file path heuristics. This will produce false positives in projects where "sensitive" file names (payments.ts, auth.ts) are the actual target of the request. Acceptable for v1 — the drift alert says "consider reviewing," not "this is wrong." The Claude Code engine provides context-aware drift detection as the upgrade path.
